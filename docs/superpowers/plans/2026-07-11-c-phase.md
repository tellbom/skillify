## Task 1: C-1 后端 — 版本中心 schema + API

**背景**：`skill_index` 目前没有 yanked 概念；`queries.py` 的 `get_versions`/`list_latest`/`search`/`leaderboard` 都没有版本状态过滤；Web 只有隐式的"最新版本"，没有版本列表/diff/yank 端点；`ForgejoClient` 没有 tree 列表方法、`Release` 也不带 `body` 字段。

**Schema（`infra/dm8-init/02-c1-version-center.sql`）**：
- `ALTER TABLE skill_index ADD COLUMN yanked BIT DEFAULT 0 NOT NULL;`（DM8 语法）。
- 对应 `models.py`：`SkillIndexEntry.yanked: Mapped[bool] = mapped_column(default=False)`（SQLite 端用 SQLAlchemy 的 Boolean 隐式类型，跟 `SkillEvent.success` 现有写法保持一致的声明风格，但这里要 `NOT NULL DEFAULT False`，不是可空的三态）。
- 变更说明**不落库**——不要新增列存储 release notes；改动说明直接从 Forgejo release body 现取。

**`src/skillify/publish/forgejo_client.py` 改动**：
- `Release` dataclass 加 `body: str = ""` 字段，`_release_from_json` 填充 `data.get("body", "")`。
- 新增 `list_tree(self, owner: str, repo: str, ref: str) -> list[dict]`：调 `GET /repos/{owner}/{repo}/git/trees/{ref}?recursive=true`，返回 Gitea/Forgejo tree API 的 `tree` 数组（每项至少含 `path`、`sha`、`type`）；404 时返回 `[]`（版本不存在时不报错，由调用方判断）。
- `tests/fake_forgejo.py` 需要扩展一个 `do_GET` 分支处理 `/api/v1/repos/{owner}/{repo}/git/trees/{ref}` 并支持 `recursive` 查询参数，返回预置的假 tree 数据（测试里通过 `fake_forgejo.state` 新增一个 `trees: dict[str, list[dict]]` 之类的存储结构，具体命名 implementer 自行决定，保持和现有 `state.raw_files` 等字段风格一致）；同时让 `create_release`/`get_release_by_tag` 返回的 fake JSON 带上 `body` 字段（默认可为空字符串）。

**`src/skillify/index/queries.py` 改动**：
- `get_versions` 保持不变（不过滤 yanked——时间线要看到全部版本并标注状态）。
- `list_latest(session, *, include_yanked: bool = False)`：默认排除 yanked 行再做"每 (namespace,name) 取 max(id)"分组——即某个 skill 的最新版本若被 yank，则该 skill 的"最新版本"回退到更早的未 yank 版本；若全部版本都被 yank，则该 skill 从列表中消失（crates.io 式行为，任务书未反对）。`include_yanked=True` 时保留旧行为（不过滤），供内部/管理用途按需调用。
- `search(session, query, *, include_yanked: bool = False)` 和 `leaderboard(session, dimension, *, include_yanked: bool = False)` 透传同一个参数给 `list_latest`。**Web 端点一律用默认值（排除 yanked）**，`include_yanked` 参数只是让函数在 SQLite 单测里可以两种行为都测到。
- 补 SQLite 单测覆盖："yank 后不出现在 list_latest/search/leaderboard，但 get_versions 仍能看到且标 yanked=True"，"skill 唯一版本被 yank 后从列表消失"，"unyank 后恢复出现"。测试放 `tests/test_index.py` 或新建 `tests/test_index_versions.py`（跟随现有平铺命名）。

**新增 `src/skillify/index/yank.py`**（跟随 `ratings.py`/`ownership.py` 的模块粒度风格）：
- `set_yanked(session, *, namespace, name, version, yanked: bool) -> SkillIndexEntry`：按 (namespace,name,version) 查唯一行，设置 `yanked` 字段，`session.commit()`；查不到抛 `VersionNotFoundError(namespace, name, version)`（新异常类，同文件定义）。
- `can_manage_version(session, *, namespace, username) -> bool`：author 或 namespace owner 才能 yank/unyank——复用 `SkillNamespaceOwner` 表查 owner；author 判断在调用方按 entry.author 比对即可，不需要这个函数额外查 author（因为 author 就在 entry 上）。命名/参数以 implementer 判断为准，只要语义是"author 或 namespace owner 才允许"。

**`src/skillify/web/app.py` / `src/skillify/web/schemas.py` 新增端点**：
- `GET /api/skills/{namespace}/{name}/versions` → `list[VersionInfo]`（新 schema：`version`, `publishedAt`, `yanked`, `releaseNotes: str | None`）。`releaseNotes` 现取 Forgejo：对每个版本调 `ForgejoClient.get_release_by_tag(org, name, f"v{version}").body`；`ForgejoClient` 不可用/未配置或该 tag 查不到时该字段填 `None`（best-effort，不让整个端点因 Forgejo 不可达而 500）。
- `GET /api/skills/{namespace}/{name}?version=X`（在现有 `skill_detail` 端点上加可选 `version: str | None = Query(default=None)` 参数）：给定 version 时返回该版本详情（含 README/tarball，即使该版本 yanked——显式指定版本要能看到/装，见任务书"yanked 仍可显式安装"），不给时保持现有"最新未 yank 版本"行为。`service.get_skill_detail` 相应加同名可选参数，内部从 `get_versions` 里挑指定版本而不是 `versions[0]`；找不到该版本时仍返回 404。
- `POST /api/skills/{namespace}/{name}/versions/{version}/yank` 和 `.../unyank`：`Depends(require_keycloak_user)`，鉴权用上面的"author 或 namespace owner"逻辑，不满足条件返回 403；成功返回 `{version, yanked}`。
- `GET /api/skills/{namespace}/{name}/diff?from=X&to=Y`：两个版本各自算 tag（`f"v{X}"`/`f"v{Y}"`），用 `ForgejoClient.list_tree` 拿两棵树，按 `path` 对比 —— 只在其中一棵出现 → `added`/`removed`；两棵都有但 `sha` 不同 → `modified`；返回 `{added: list[str], removed: list[str], modified: list[str]}`。Forgejo 未配置或版本查不到时返回 4xx（400/404 均可，implementer 判断哪个更贴合现有风格）。**不落库，纯计算**。
- CLI 版本安装（`skillctl install ns/name@version`）：确认 `src/skillify/cli/install_cmd.py::run_install` 经 `parse_identifier`/`install_with_dependencies` 已支持 `@version` 语法（不经过 DM8 索引，直接查 Forgejo release，因此 yanked 状态天然不影响 CLI 安装）——**只需要补一个 SQLite/fake_forgejo 单测确认这条路径行为符合预期**，不需要改代码（除非阅读后发现确实有缺口，此时按最小改动修）。

**产出物清单**：`infra/dm8-init/02-c1-version-center.sql`、`models.py`（yanked 列）、`queries.py`（过滤参数）、新 `yank.py`、`forgejo_client.py`（body 字段 + list_tree）、`app.py`/`schemas.py`/`service.py`（新端点）、`tests/fake_forgejo.py`（tree 端点 + release body）、对应 SQLite 单测。

## Task 3: A-2 — Forgejo 草稿发布 + 幂等续传（后端，无 schema）

**背景**：用户已裁决在 C-2 范围内补齐这块。目标：让 `publish_skill_dir` 对同一个本地 `skill_dir` 的重复调用天然可续传（CLI 重试、webhook 重投都受益），不引入新表——这一层解决的是"Forgejo release 创建到 asset 上传之间中途失败"的可恢复性；C-2（Task 4）另外解决"Web 上传失败后如何在前端看到并触发重试"的可见性问题，两者配合才是完整的 A-2。

**`src/skillify/publish/forgejo_client.py` 改动**：
- `create_release(...)` 增加 `draft: bool = False` 参数，透传进请求 JSON 的 `"draft"` 字段（保持默认值 `False`，不改变现有调用方行为除非显式传 `draft=True`）。
- `Release` dataclass 加 `draft: bool = False` 字段，`_release_from_json` 填充 `data.get("draft", False)`。
- 新增 `update_release(self, owner, repo, release_id, *, draft: bool | None = None, body: str | None = None) -> Release`：`PATCH /repos/{owner}/{repo}/releases/{release_id}`，只在参数非 None 时把对应字段放进请求体（避免覆盖未传字段）；非 200 抛 `ForgejoError`。
- `tests/fake_forgejo.py`：`do_POST` 里 `create_release` 分支要接住并存储 `draft` 字段（默认 `False`）；新增 `do_PATCH` handler 支持 `/api/v1/repos/{owner}/{repo}/releases/{id}`，更新对应存储里的 `draft`/`body` 字段并返回更新后的 release JSON。

**`src/skillify/publish/publisher.py::publish_skill_dir` 改动**（核心逻辑）：
- 查 `existing = client.get_release_by_tag(org, repo, tag)` 后：
  - `existing is not None and not existing.draft` → 保持现状，抛 `AlreadyPublishedError`（不可变版本策略不变）。
  - `existing is not None and existing.draft` → **续传**：复用这个 release（不新建），跳过已经在 `existing.assets` 里出现（按文件名匹配）的资产，只上传缺失的。
  - `existing is None` → 用 `draft=True` 创建 release（而不是现在的 `draft=False`），再上传全部资产。
  - 三个分支上传资产完毕后，统一调 `client.update_release(org, repo, release.id, draft=False)` 把草稿转正——这一步失败会向上抛 `ForgejoError`，调用方（CLI/webhook/web upload）看到的错误行为不变，只是现在下一次重试会命中"续传"分支而不是撞见 `AlreadyPublishedError`。
- 索引写入（`_index_release`）逻辑不变（本来就是 best-effort、`upsert_release` 本来就幂等）。
- **测试**：`tests/test_publish_index_integration.py`（或新建一个聚焦文件，跟随现有测试文件按功能而非任务编号命名的惯例）要覆盖：①正常发布仍然一次成功（回归）；②模拟"资产上传到一半中断"（比如 mock `upload_release_asset` 在第二次调用时抛异常）后再次调用 `publish_skill_dir`，第二次应该跳过已上传资产、只传剩下的、最终 release 变为非 draft；③已存在非 draft release 时仍然抛 `AlreadyPublishedError`（回归）。

**产出物清单**：`forgejo_client.py`（draft/update_release/Release 字段）、`publisher.py`（续传逻辑）、`tests/fake_forgejo.py`（draft/PATCH 支持）、新增/扩展测试。**无 schema 改动**。

## Task 4: C-2 后端 — 我的 Skill / 我的 Namespace / 失败发布可见性

**依赖**：Task 3 的 draft 续传机制已存在。

**Schema（`infra/dm8-init/03-c2-my-skills.sql`）**：
- 新表 `skill_publish_jobs`：记录"谁在什么时候尝试发布了哪个 namespace/name/version，结果如何"，这是"我的失败发布"列表的数据来源（不用扫描 Forgejo 全量仓库找 draft release，那样在多 namespace 场景下不可行）。
- 列：`id`（PK IDENTITY）、`namespace`、`name`、`version`（均 VARCHAR(64) NOT NULL）、`initiator VARCHAR(255) NOT NULL`（发起人 Keycloak 用户名）、`status VARCHAR(16) NOT NULL`（`'succeeded' | 'failed'`）、`error_message VARCHAR(2000)`（可空）、`created_at`/`updated_at TIMESTAMP WITH TIME ZONE NOT NULL`。`UniqueConstraint(namespace, name, version)`——同一个版本号的发布尝试只保留最新一条状态（重试会更新同一行的 status/error_message/updated_at，不是插入新行）。
- 对应 `models.py`：新增 `SkillPublishJob` 模型，字段/约束与上面列级一致。

**新增 `src/skillify/index/publish_jobs.py`**（跟随 `ingest.py`/`ownership.py` 的 SAVEPOINT-upsert 风格，因为这是"同一 (namespace,name,version) 可能被并发/重复写"的场景）：
- `record_job_result(session, *, namespace, name, version, initiator, status, error_message, at) -> SkillPublishJob`：insert-or-update（按 unique constraint），跟 `upsert_release` 的 try/except IntegrityError 模式一致。
- `list_my_failed_jobs(session, initiator) -> list[SkillPublishJob]`：`status == 'failed'` 且 `initiator == 当前用户`，按 `updated_at desc`。
- `list_my_jobs(session, initiator) -> list[SkillPublishJob]`（全部状态，供"发布结果"列表用，不只是失败的）。

**`src/skillify/web/upload_service.py::handle_upload` 改动**：
- 在能解析出 `namespace`/`declared_name`（此时还不一定有 `version`——需要读一下 `manifest.get("version")`，若拿不到就用空字符串占位，不阻塞整个流程；validate 阶段之前的失败——比如 zip 都解不开——不记录 job，因为那时候连 namespace/name 都不知道，不是"发布失败"而是"上传请求本身不合法"）之后，用 `try/except` 包住剩下的流程（namespace 认领 + `publish_skill_dir`）：成功则 `record_job_result(..., status="succeeded", error_message=None)`，任何异常都 `record_job_result(..., status="failed", error_message=str(exc))` 后**重新抛出**原异常（`app.py` 现有的 exception→HTTP 状态码映射逻辑不变，job 记录是旁路副作用，不能改变原有 API 错误行为）。这一步需要 `index_db_url` 才能记录——沿用 `_index_release` 的 best-effort 精神：job 记录本身失败（比如索引库不可达）不应该掩盖或替换原始发布异常，用内层 `try/except Exception: pass` 包住 job 写入调用本身。

**`src/skillify/index/queries.py` 或新增聚合函数**（放哪个文件跟随现有粒度，倾向新建 `src/skillify/index/my_skills.py`）：
- `list_my_skills(session, username) -> list[SkillIndexEntry]`：`author == username` 的最新版本集合（复用 `list_latest` 的分组逻辑再按 author 过滤，或直接查 `skill_index` 按 author 分组——implementer 判断怎么写更贴合现有 `list_latest` 实现，避免重复造轮子）。
- `list_my_namespaces(session, username) -> list[SkillNamespaceOwner]`：`owner_username == username` 的行。
- `my_usage_stats(session, username) -> dict`：复用 `events.py::install_counts` 按 author 过滤汇总（installs 总数等），具体聚合维度不需要复杂，够"我的 Skill"页面展示总安装次数即可，不做过度设计。

**`src/skillify/web/app.py`/`schemas.py` 新增端点**（均 `Depends(require_keycloak_user)`，身份取 `preferred_username`/`sub`）：

## Task 9: C-4 后端 — 搜索下沉 SQL

**背景**：这是任务书里明确标注"最谨慎处理"的一步——`queries.search()` 目前是纯 Python 子串匹配 + 全表加载，要改成 SQL 层分页/过滤/排序，同时不能引入任何手写 SQL 或方言相关写法（DM8 分页与 `LIKE` 大小写语义跟 Postgres/SQLite 有差异，必须全部走 SQLAlchemy 表达式）。

**改动**：
- `queries.py::search` 签名扩展为 `search(session, query: str | None, *, namespace: str | None = None, author: str | None = None, tags: list[str] | None = None, sort: str = "updated", page: int = 1, page_size: int = 20, include_yanked: bool = False) -> tuple[list[SkillIndexEntry], int]`（返回结果页 + 总数，供前端分页用）。
- 必须用 `.offset()`/`.limit()` 做分页；`func.lower(SkillIndexEntry.name).like(func.lower(...))` 或 SQLAlchemy 的 `.ilike()`（如果目标方言都支持就用 `.ilike()`，否则显式 `func.lower(...).like(...)` 两边都转小写，规避 DM8/Postgres 大小写语义差异）做名称/描述过滤——**绝不能手写 `text("...")` SQL 字符串**。
- `namespace`/`author` 过滤：简单 `.where(SkillIndexEntry.namespace == namespace)` 等值匹配。
- `tags` 过滤：**保留在应用层**（任务书明确交代 DM8 JSON 函数方言差异大，首版不做，不要引入标签规范化表）——即先用 SQL 过滤 namespace/author/文本query 缩小结果集，再在 Python 里对 `tags` 字段做交集判断，最后在 Python 层分页？不行——分页必须在 SQL 层做否则失去意义。正确做法：如果 `tags` 参数给了值，就放弃 SQL 端分页，退化成"SQL 只做 namespace/author/文本过滤+排序，取全部匹配行到 Python 里再按 tags 过滤，然后手动切片分页"——这是任务书允许的"首版仍在应用层"过滤的自然结果，**在代码注释和任务报告里显式写清这个退化路径**，不要假装它是全 SQL 分页。
- `sort` 取值 `"installs" | "rating" | "updated"`：`updated` 直接 `.order_by(SkillIndexEntry.published_at.desc())`；`installs`/`rating` 需要 JOIN/子查询聚合（`skill_events`/`skill_ratings`）再排序——用 SQLAlchemy 的子查询/`outerjoin` 表达式实现，参考 `list_latest` 现有"max(id) 分组子查询"的写法风格。这是本任务技术难度最高的部分，implementer 要先写通 SQLite 单测再收工，不要为了赶进度手写子查询绕过 ORM 层。
- 先用"每 (namespace,name) 取最新未 yank 版本"的既有子查询作为基表（复用/重构 `list_latest` 的子查询逻辑，不要重复造轮子——可以把 `list_latest` 内部的"最新版本子查询"抽成一个私有辅助函数供 `search` 复用）。
- **行为等价性**：`search(session, query)`（不传新参数）在只有 `query` 时的返回顺序/内容跟旧版本 `search()` 的历史测试断言应该保持一致或者显式更新那些断言并在报告里说明差异原因（任务书 §C-4 最后一条要求）。

**`app.py::search_skills`/`list_skills` 改动**：加 `namespace`/`author`/`tags`（逗号分隔字符串或多值 query）/`sort`/`page`/`pageSize` 查询参数，透传给新版 `search`，响应体加分页信息（`total`/`page`/`pageSize`，具体包装成什么 schema——是否新建 `SearchResult` 分页壳——implementer 判断，跟 `SkillSummary` 现有风格保持一致）。

**Schema（`infra/dm8-init/05-c4-search-indexes.sql`，仅在真的加了新排序/过滤列索引时创建）**：如果实现过程中发现某个过滤列（如 `author`）需要索引来避免全表扫（虽然内网数据量小，但顺手加不复杂），才创建这个文件，只用 `CREATE INDEX`；如果不需要就不创建这个文件（序号占位留空，任务书原话）。

**测试**：改造/新增 `tests/test_index.py` 里 `search` 的用例，覆盖分页、namespace/author 过滤、tags 应用层过滤退化路径、三种 sort 的排序正确性、yanked 排除。
